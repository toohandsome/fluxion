# Fluxion 正式需求文档

## 1. 文档目的

本文档用于明确 Fluxion 一期的正式需求边界，作为产品、研发、测试和实施阶段的统一基线。

## 2. 项目定义

Fluxion 是一个基于 Spring Boot 的可视化流程编排系统，支持用户通过图形化方式设计业务流程，并将流程发布为 HTTP 接口或定时任务执行。

一期产品同时支持两种使用形态：

1. 嵌入式形态
   以 `Spring Boot Starter` 的方式集成到业务系统，核心能力与 `Server` 等价（设计、发布、执行、监控、治理）；默认不自启独立 Web 端口，可按宿主需要启用 Web/API 与前端页面能力。
2. 平台形态
   基于 `Starter` 提供开箱即用的独立部署形态，默认启用 Web/API 暴露与前端页面集成能力，并以独立进程运行。

## 3. 设计边界

一期采用以下边界原则：

1. 主流程必须是 DAG，不允许通过图回边表达循环。
2. 平台默认执行语义为 `at-least-once`。
3. 不提供分布式事务，由重试、补偿策略兜底。
4. 一期不开放插件热加载、前端插件、流式处理、断点调试。
5. 一期数据库只承诺支持 `pg`。

## 4. 目标用户

1. 平台管理员
   负责资源配置、流程发布、任务启停、系统治理。
2. 流程设计人员
   负责流程设计、调试、发布。
3. 业务调用方
   通过 HTTP 或定时任务触发流程执行。

## 5. 一期功能范围

### 5.1 流程设计

系统应提供基础可视化流程设计器，支持：

1. 创建、编辑、删除流程。
2. 拖拽节点并建立连接关系。
3. 节点参数配置。
4. 保存草稿和发布版本。
5. 基础结构校验。

一期至少支持以下校验：

1. DAG 校验。
2. 必填参数校验。
3. 节点输入输出连通性校验，包括所有节点必须位于同一个连通分量内。
4. 所有节点必须同时满足“可从至少一个起始节点到达”且“可到达至少一个结束节点”。
5. 发布前模型编译校验。

### 5.2 流程版本管理

系统应支持：

1. 流程定义、流程草稿与流程版本分离。
2. 一个流程定义同一时间只允许存在一个草稿。
3. 草稿不占正式版本号，发布时才分配新的递增 `version_num`。
4. 发布后形成不可变快照，正式版本只包含发布和归档两种状态。
5. 查询当前生效版本、历史版本和当前草稿。

### 5.3 执行引擎

一期执行引擎应支持：

1. 顺序执行。
2. 条件分支执行。
3. 有限并发执行。
4. 变量上下文传递。
5. 节点超时控制。
6. 节点失败重试。
7. 流程失败终止。
8. 流程显式最终输出映射。

一期暂不包含：

1. 子流程。
2. 脚本节点。
3. 流式处理。
4. 人工任务。

### 5.4 基础节点

一期至少提供以下节点类型：

1. 日志节点
2. 变量设置/转换节点
3. 条件判断节点
4. HTTP 请求节点
5. 数据库查询节点
6. 数据库更新节点

### 5.5 HTTP 发布

系统应支持将流程发布为 HTTP 接口，能力包括：

1. 配置请求路径和方法。
2. 提取 path、query、header、body 参数。
3. 参数校验与类型转换。
4. 响应映射。
5. 同步和异步触发模式。
6. 基础认证方式和限流配置。
7. 运行时路由冲突检测；若装配时发现与宿主已有路由冲突，则端点自动转为 `DISABLED` 并跳过注册。

### 5.6 定时任务

系统应支持基于 Quartz 的流程调度，能力包括：

1. 配置 Cron 表达式。
2. 启停任务。
3. 配置错失策略，至少包含 `DO_NOTHING`、`FIRE_ONCE_NOW`、`CATCH_UP_BOUNDED`。
4. 查询任务最近和下次触发时间。
5. 记录触发流水。
6. 配置调度侧等待超时 `waitTimeoutMs`。
7. 查询触发流水列表与详情。

### 5.7 运行监控

系统应支持：

1. 查询流程实例。
2. 查询节点执行记录。
3. 查看输入输出快照。
4. 查看错误摘要和异常堆栈。
5. 通过业务键、流程编码、时间范围检索执行记录。
6. 对 `SKIPPED` 节点支持查看跳过原因；未真实执行的节点不要求有输入输出快照。

### 5.8 治理能力

一期必须具备以下治理能力：

1. 资源连接与密钥托管。
2. 操作审计。
3. 节点超时与重试策略。
4. HTTP 接口基础认证和限流配置。
5. 调度任务并发和重入控制。

## 6. 一期默认产品约定

以下约定作为一期正式需求的默认落地边界。如果后续没有新的需求决议，研发和测试将按此作为基线实施。

### 6.1 实现范围

1. 一期交付基础前后端闭环能力，包含基础可视化设计器和后端执行能力。
2. 一期按单租户实现，`tenant_id` 保留为扩展字段，默认使用固定租户值。
3. 一期目标是“最小可用闭环”，不包含插件、流式处理、复杂调试等高阶能力。

### 6.2 节点范围

一期默认只实现以下节点：

1. 日志节点
2. 变量设置/转换节点
3. 条件判断节点
4. HTTP 请求节点
5. 数据库查询节点
6. 数据库更新节点

### 6.3 前端交付边界

一期前端默认交付以下基础能力：

1. 流程列表与基础管理页面
2. 流程画布编辑
3. 节点拖拽、连线与删除
4. 节点参数配置表单
5. 草稿保存与发布操作
6. 流程实例与节点执行记录查看
7. 表达式自动补全
8. 节点引用选择器
9. 节点与流程输出默认模板
10. 调度触发流水列表与详情查看

一期前端暂不包含：

1. 前端插件动态加载
2. 高级调试器
3. 模板市场
4. 复杂协同编辑

### 6.4 发布与触发语义

1. 一期管理接口与运行时结果查询接口统一采用同一响应包结构；已匹配到的运行时 HTTP 发布端点允许通过 `responseConfig` 自定义整个 JSON envelope。
2. HTTP 发布响应默认使用统一 JSON 包装；当 `responseConfig.envelopeMode = CUSTOM_JSON` 时，端点业务响应可直接返回自定义 JSON body。
3. 同步 HTTP 接口在统一包装模式下，未配置 `successDataMapping` 时，在响应包的 `data` 中默认返回 `{ "instanceId": instance.instanceId, "status": instance.status, "result": flow.output }`。
4. 异步 HTTP 接口在统一包装模式下，未配置 `runningDataMapping` 时，在响应包的 `data` 中默认返回 `instanceId`、实例状态和结果查询地址。
5. 一期所有统一包装体响应都必须返回 `requestId`；运行时 HTTP 自定义 envelope 时可不返回。
6. 一期调度采用单机 Quartz，重入策略只支持 `FORBID` 和 `ALLOW`。
7. `reentryPolicy = FORBID` 时，`maxConcurrency` 必须固定为 `1`。
8. 一期调度不提供独立排队语义；并发门禁不通过时直接拒绝本次触发并记录触发流水。
9. 调度侧等待字段统一命名为 `waitTimeoutMs`；其含义是“调度侧等待实例进入终态的最长时间”，超时只表示停止等待，不等同于实例失败。
10. 一期推荐的补跑追赶策略为 `CATCH_UP_BOUNDED`；未显式提供 `catchUpConfig` 时，默认采用“最近 1 小时、最多 10 次、最老优先”。
11. 对外触发流水列表默认只展示汇总状态 `summaryStatus`，推荐展示文案为“已受理、并发拒绝、等待超时、实例成功、实例失败”。
12. 触发流水详情页或详情抽屉再展开 `dispatchStatus`、`waitStatus`、`instanceStatusSnapshot` 与错误信息等内部状态字段。

### 6.5 数据库与安全边界

1. 一期正式支持数据库固定为 `pg`。
2. 不提供分布式事务，由重试和补偿策略兜底。
3. 一期默认按“内网管理系统 + 基础认证”场景建设。
4. 一期不强依赖完整 `Spring Security` 体系。

### 6.6 流程生命周期规则

1. 一个流程定义在同一时间只允许存在一个可编辑草稿。
2. 草稿独立于正式版本存储，允许反复保存和校验，但草稿本身不可直接执行。
3. 草稿不占正式版本号，草稿阶段通过 `draft_revision` 或更新时间控制并发。
4. 草稿可记录 `base_version_id`，表示当前编辑基于哪个已发布版本继续展开。
5. 发布动作必须基于当前草稿完成校验和编译，并生成新的不可变版本快照。
6. 发布成功时分配新的正式版本号，规则为当前 `latest_version_num + 1`；首个发布版本号固定为 `1`。
7. 一个流程定义同一时间只允许有一个“当前生效”的已发布版本，由 `active_version_id` 指向。
8. 新版本发布成功后，旧的当前生效版本自动转为归档状态。
9. 发布成功后，当前草稿继续保留，并将 `base_version_id` 更新为最新发布版本，作为后续编辑起点。
10. 归档版本不可修改，但历史实例仍可查询，固定引用该版本的运行记录继续保持可追溯。
11. 流程删除采用逻辑删除，不做物理删除。
12. 删除流程前，必须先下线全部 HTTP 发布端点，并暂停或禁用全部调度任务。
13. 已删除流程不可再编辑、发布或新触发，但历史版本、实例、审计日志必须保留。
14. HTTP 端点下线后，不再接收新的外部请求，但不影响已启动实例继续执行。
15. 调度任务暂停后，不再产生新的触发记录，但不影响已启动实例继续执行。

## 7. 非功能要求

### 7.1 性能

1. 支持 I/O 型流程并发执行。
2. 支持虚拟线程作为默认执行载体。
3. 执行能力受并发上限和超时控制约束。

### 7.2 稳定性

1. 已发布版本执行语义稳定，不因草稿修改而变化。
2. 执行日志写入与大字段快照应保持本地事务一致性。
3. 系统应具备日志清理和归档策略。

### 7.3 安全性

1. 资源敏感信息不得明文回显。
2. 平台关键操作必须记录审计日志。
3. 补偿策略在一期只做声明与审计，不做自动补偿编排。

## 8. 一期明确不做

1. PF4J 插件热加载
2. 前端插件动态加载
3. 模板库
4. 子流程
5. Groovy 脚本节点
6. Redis 节点
7. 流式处理
8. 断点调试
9. 多数据库兼容

## 9. 一期交付物

1. 一套可运行的流程编排后端服务
2. 一套基础前端管理页面与可视化设计器
3. 一套基础管理接口与运行时接口
4. 一套可嵌入业务系统的 Starter
5. 一期数据库 DDL
6. 一期 Maven 模块骨架
