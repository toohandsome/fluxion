# 历史完整版

Fluxion 织流 —— 可视化流程编排的织机，让业务逻辑如丝般顺滑流淌。

可视化流程编排组件/平台（Spring Boot ）需求规格
1. 项目概述
构建一个基于 Spring Boot 的可视化流程编排组件（Starter），允许用户通过拖拽方式设计业务逻辑流程，将流程发布为 HTTP 接口或定时任务，并支持插件化扩展、分布式调度、多数据库持久化及完善的监控运维能力。旨在提供类似 Apache NiFi 或 Odin 的低代码集成体验，降低开发复杂度，提升业务响应效率。
流程执行默认语义为 at-least-once；
所有外部副作用节点需提供补偿策略;
脚本节点和插件能力仅对受信管理员开放，不作为强隔离安全边界。

2. 核心功能需求
2.1 可视化流程设计
流程图编辑器：提供基于 Web 的拖拽式设计界面，支持创建、编辑、删除流程节点，连接节点构成有向图（支持条件分支和循环）。

节点类型库：

内置基础节点：HTTP 请求、数据库查询、Redis 操作、日志记录、条件判断（if/else）、循环（for/while）、脚本执行、“子流程节点”、“数据库游标查询”等。

节点配置界面：每个节点提供独立的配置面板，支持定义输入参数、输出参数、连接属性、是否记录日志、重试策略等；若无特定界面，则展示通用配置表单（基于 JSON Schema 自动生成）。

流程设计器校验：

环路检测：明确检测范围为主流程图，主图和子图均强制为 DAG，循环语义通过专用循环节点封装实现，而非图中的回边。对于循环节点，其内部子图同样需进行环路检测，环路检测分两个层级：主图层级：将循环节点视为原子节点，仅检测其外部连接是否成环。子图层级：展开循环节点内部，强制要求内部子图为 DAG（有向无环图）。跨层级规则：禁止从循环内部节点直接连接到循环外部节点（必须通过循环节点的出口）

连接逻辑校验：循环节点必须有至少一个输入和一个输出（循环结束后继续执行），内部子图需有明确的开始和结束。

必填参数检查：节点配置中标记为必填的字段若未填写，保存时给出错误提示。

流程参数化：支持在流程中定义变量，并在节点间传递；变量可来源于外部输入（HTTP 请求参数、定时任务触发参数）或其他节点输出。

2.2 插件化扩展机制
自定义节点插件：

允许开发者编写自定义节点（包含后端逻辑和前端配置界面），打包上传至组件。

后端插件支持热加载（通过pf4j 实现），前端插件支持动态加载 ：ES Modules (ESM) 动态导入 或 Vue 3 的异步组件。
严格限制插件的权限，尽量不要让插件内部的类注册为 Spring Bean，而是让插件实现纯粹的 Java 接口（如 PluginNodeRunner）。引擎在执行时，通过 PF4J 的 PluginManager.getExtensions() 获取实例并调用。
这样可以避开 Spring 上下文动态卸载的噩梦，只利用 PF4J 的类隔离和热加载特性。

插件元数据管理：存储插件名称、版本、作者、依赖关系等，支持版本控制。


2.3 流程执行引擎
执行模型：流程基于图结构执行，支持顺序、并行、条件分支和循环。

条件分支：使用“条件判断节点”，根据表达式结果选择后续分支（真/假），分支输出连接到不同节点。

循环：
for-each：遍历集合（如数组、列表），每次迭代执行内部子图，循环变量可通过表达式 ${loop.current} 访问。
while：条件为真时重复执行内部子图。
循环结束后，可配置输出聚合方式（如收集每次迭代的结果为列表）。
循环内部子图必须是无环的，且不能包含等待人工任务的节点（避免无限期挂起）。

流程实例状态机：每个流程实例具有明确的生命周期状态，包括：

创建：流程实例已生成，但尚未开始执行。

启动：已触发执行，正在初始化。

执行中：流程正在运行，至少有一个节点处于活动状态。

成功：流程所有节点正常执行完毕，达到结束状态。

失败：流程执行过程中遇到不可恢复的错误，终止执行。

并发执行：

当流程中存在多个并行分支（如分支节点后连接多个无依赖节点）时，引擎会并发执行这些分支，充分利用线程池资源。

并发执行充分利用 Java 21+ 虚拟线程（Virtual Threads），为每个并行分支创建独立的虚拟线程，显著降低线程创建和上下文切换开销，提升系统吞吐量。虚拟线程由平台线程（Platform Threads）调度，当分支执行 I/O 操作（如 HTTP 请求、数据库查询）时自动让出载体线程，实现高效阻塞。

并发分支执行完成后，通过汇聚节点（如 join）同步结果。

并发数受线程池配置限制，避免资源耗尽。

2.4 流程发布为 HTTP 接口
动态接口生成：将设计好的流程动态注册为 Spring MVC 的 HTTP 端点，支持自定义 HTTP 方法（GET/POST/PUT/DELETE 等）、URL 路径、请求参数类型。

参数绑定与引用：

支持从请求的 path、query、header、body（JSON/XML/Form/Raw）中提取数据，并通过表达式语言（如 ${request.query.id}、${request.body.user.name}）和  JSON Path（$.user.name） 在流程节点中引用。

支持请求体校验、类型转换。

响应处理：流程执行结果可转换为指定格式（JSON/XML/文本），支持自定义 HTTP 状态码和响应头。（自定义优先，不受全局响应格式影响）

2.5 定时任务调度
集成 Quartz 作为调度引擎：使用 Quartz 管理定时任务的持久化、分布式调度、Cron 表达式动态修改。

任务定义：可将流程绑定为定时任务，设置 Cron 表达式、时区、任务参数等。

动态调整：支持运行时修改 Cron 表达式，无需重启应用；支持暂停、恢复、立即触发一次任务。

分布式支持：基于数据库 实现 Quartz 集群模式，确保任务在分布式环境中不重复执行。

2.6 数据持久化
多数据库兼容：支持 MySQL、PostgreSQL、SQL Server、Oracle、H2 等主流关系型数据库，使用  MyBatisPlus 抽象存储层，并通过 Flyway 管理数据库脚本。

存储内容：

流程定义（包括流程图 JSON、节点配置、版本信息）

流程实例（执行记录、状态、输入输出数据快照）

定时任务定义与状态

插件元数据

系统日志与监控数据

流程版本管理：

每次修改流程定义生成新版本，旧版本流程实例可继续执行完成或按策略迁移。

支持版本回滚、版本对比。

2.7 错误处理与重试机制
节点级错误处理：每个节点可配置重试策略（最大重试次数、重试间隔、退避算法）及失败后的路由（如“错误分支”或“继续执行”）。

全局错误处理：定义全局异常处理器，捕获未处理异常，记录错误日志。


2.8 监控与日志
全链路日志：记录每个流程实例的执行轨迹，包括节点启动/结束时间、输入输出数据（可脱敏）、异常堆栈。

可视化监控：

流程实例列表：支持按状态（运行中、成功、失败等）、时间范围查询。

实例详情：展示 DAG 执行图，高亮显示成功/失败节点，并可查看节点详细日志。

执行轨迹可视化：以时序图或流程图形式展示流程执行的详细路径，标注每个节点的输入输出数据流转，便于追溯问题。

统计指标：包含流程执行次数、耗时、错误率等。


2.9 数据转换与脚本支持
内置数据转换节点：提供 JSON 转 XML、XML 转 JSON、字段映射、数据聚合/拆分等功能。

脚本节点：支持 Groovy 等脚本语言，允许用户在流程中编写自定义逻辑（前端使用monaco 做语法提示），并可引用上下文变量。

表达式语言：内置 SpEL 或自定义 EL，用于条件判断、变量赋值、参数提取等场景。

2.10 线程池管理
传统线程池：适用于 CPU 密集型任务，使用平台线程（Platform Threads）。

虚拟线程池（可选）： 基于 Executors.newVirtualThreadPerTaskExecutor() 提供虚拟线程执行器，适用于大量 I/O 密集型任务。虚拟线程由 JVM 管理，可支持极高并发度，线程资源占用极低。用户可根据节点类型（如数据库查询、HTTP 调用）选择使用虚拟线程执行器，或在全局层面启用虚拟线程模式。


隔离机制：可为不同优先级或租户的流程分配独立线程池，避免相互干扰。

异步处理：HTTP 接口触发流程时支持同步或异步模式，异步模式下立即返回任务 ID，后续可通过查询接口获取执行结果。

可为流处理任务配置隔离的线程池，并支持动态调整并行度。



2.11 流程调试与测试
流程测试功能：在流程发布前，允许用户输入测试数据（模拟 HTTP 请求参数、定时任务触发参数等），执行流程，查看每个节点的中间输出和最终结果，便于验证逻辑正确性。

断点调试（可选）：支持在节点上设置断点，单步执行，观察变量变化。

2.12 流程模板库
模板管理：支持将常用业务流程保存为模板，模板包含完整的节点配置和连接关系，但不包含具体业务参数（如具体的 URL、SQL 语句等）。

模板版本控制：模板可进行版本管理，每次修改保存为新版本，支持版本回滚和对比。

模板导入导出：支持以 JSON 格式导出模板，并可从 JSON 文件导入模板，方便模板分享和迁移。

基于模板创建流程：用户可从模板库中选择模板，一键生成新流程，然后补充具体参数和调整细节，提高开发效率。

2.13 分布式支持
集群部署：组件支持多节点部署，通过共享数据库或 Redis 实现流程实例锁、定时任务协调。

分布式锁：确保关键操作（如流程版本发布、任务触发）的原子性。


2.14 子流程（Sub-Process）
目标：支持将一组节点封装为可复用的子流程，提高复杂流程的模块化程度和设计效率。

2.14.1 子流程定义
嵌入式子流程：在流程设计器中，允许选中一组节点（构成连通子图）并将其折叠为一个“子流程节点”。该节点内部包含完整的流程图，双击可展开编辑。

独立子流程（流程调用）：支持将已有的独立流程定义作为子流程被其他流程调用。调用方式类似于函数调用，可传递输入参数并接收返回值。

2.14.2 子流程节点配置
引用选择：若使用独立子流程，需从已发布的流程列表中选择一个流程作为子流程。

参数映射：配置子流程的输入参数来源（可来自父流程的上下文变量、常量或表达式），并指定子流程的输出结果如何存储到父流程上下文中（如 ${parent.subResult}）。

执行模式：支持同步（阻塞等待子流程完成）或异步（触发子流程后继续执行父流程，需有汇合机制）。默认采用同步模式以保证数据一致性。

版本策略：子流程可指定使用“最新版本”或“固定版本”，当子流程更新时，父流程的行为可控。

2.14.3 设计器增强
嵌套编辑：双击子流程节点进入子流程画布，同时保留面包屑导航，方便返回父流程。

参数透传：子流程内部节点可直接引用父流程传递的变量，表达式语法支持 ${parent.xxx} 或 ${xxx}（自动向上查找）。

校验规则：

禁止循环调用（流程A调用B，B又调用A）。

子流程必须有明确的开始和结束节点（或出口）。

子流程内部同样遵循主流程的环路检测规则。

2.14.4 执行引擎支持
上下文隔离：子流程执行时拥有独立的运行时上下文，继承父流程的变量，但内部变量修改默认不反向影响父流程（除非通过输出参数显式传回）。

状态管理：子流程实例作为父流程实例的嵌套子实例，在监控界面可展开查看父子关系。

并发限制：子流程的并发执行受父流程所在线程池管理，避免无限递归导致资源耗尽。

2.15 大结果集分批/游标式处理
目标：支持对大规模数据集进行流式、分批处理，避免内存溢出，提升吞吐量。

2.15.1 流式数据源节点
新增以下节点类型，用于产生数据流：

数据库游标查询节点：以游标方式（如 MyBatis Cursor）读取数据库中的大量记录，每次迭代读取一批数据（如每批100条），逐批向下游发送。

分页API调用节点：自动处理API的分页逻辑，每次请求一页，将当前页的数据列表发出。

2.15.2 流式处理节点
数据转换节点（流式版）：支持对每一条数据进行转换、过滤，类似于Java Stream的map、filter操作。配置时通过表达式或脚本定义转换逻辑。

分批存储节点：将流中的数据累积到一定数量后，执行批量插入数据库或批量调用HTTP接口。

2.15.3 循环节点的流式扩展
For-Each 增强：允许选择“流式迭代”模式。当上游传入一个大型集合（如数据库游标）时，引擎不会一次性将全部数据加载到内存，而是逐条（或逐批）拉取并执行循环体，执行完一条后释放相关资源。

并行流式处理：支持配置并行度，对数据流进行分片并发处理，利用多线程加速。并行流式处理同样可基于虚拟线程实现，为每个分片分配一个虚拟线程，进一步提升处理效率。

2.15.4 流量控制

动态限流：可配置每秒最大处理条数。

2.15.5 数据流监控
流处理指标：监控数据流入速率、当前积压量、处理延迟、成功/失败条数。

死信队列：处理失败的数据可路由到“死信节点”进行记录或重试。


3. 非功能性需求
3.1 性能

线程池饱和时，应有明确的拒绝策略（如丢弃、阻塞、调用者运行）。
借助虚拟线程，可支持每秒处理数千个以 I/O 为主的简单流程实例，显著降低延迟并提高资源利用率。具体性能取决于硬件和节点复杂度。

支持每秒处理数百个简单流程实例（取决于硬件和节点复杂度）。


3.2 可扩展性
核心模块与具体节点实现解耦，便于添加新节点类型。

插件机制需支持依赖管理和版本隔离，避免类冲突。

3.3 易用性
提供直观的 Web 设计器，降低学习成本。

支持流程导入/导出（JSON 格式），便于备份和迁移。

完善的文档和示例，帮助开发者快速上手。

3.4 
用户可以直接启动或者依赖starter集成到自己项目内部进行使用

4. 技术实现建议
后端：Spring Boot 3.5，Spring MVC，MybatisPlus,jackson，Quartz，Groovy（脚本）、pf4j 。
利用 Spring Boot 3.5 对虚拟线程的自动配置，可启用虚拟线程执行器处理异步请求和并行任务。通过 spring.threads.virtual.enabled=true 开启虚拟线程支持。

流程引擎中并发分支的执行可直接使用虚拟线程，避免显式线程池管理。对于 CPU 密集型任务，仍可使用传统平台线程池。


前端：Vue 3 ,monaco(在线脚本编辑)，流程图库可使用AntV X6，动态表单使用 Formily ，前端插件  ES Modules (ESM) 动态导入 或 Vue 3 的异步组件。

数据库：支持多方言，通过  MybatisPlus 实现。

部署：jdk21 ，打成 Spring Boot Starter，用户只需引入依赖并配置数据源即可启用。


一期：做“最小可用平台”。包含基础设计器、流程定义/版本、核心引擎、HTTP/DB/日志/条件/变量/转换节点、HTTP 发布、基础定时任务、实例与节点执行监控、PostgreSQL、H2支持、基础权限、超时重试。
二期：做“可企业内落地”。包含子流程、模板库、异步调用结果查询、取消/重放、Quartz 集群、Redis 节点、脚本节点、发布环境管理、告警通知、SQL Server/ Oracle/Mysql 支持、后端扩展 SPI 的规范化。
三期：做“高级生态与大规模能力”。包含 PF4J 热加载、前端插件、流式/游标/分页批处理、死信队列、动态限流、断点调试、多租户配额与资源隔离、插件签名与兼容性治理。
